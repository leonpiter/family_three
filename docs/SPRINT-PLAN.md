# Спринт-план: Семейное генеалогическое древо

Проект: веб-сервис семейного древа для одной семьи — бесконечная доска с круглыми фото-нодами родственников (DnD, связи «родитель—ребёнок» и «супруги»), правый сайдбар с профилем человека и фотоальбомом. Регистрация открыта, доступ — после одобрения админом.

Стек: React 19 + Vite 7 + TypeScript + Tailwind 4 + @xyflow/react 12 (доска) + zustand; бэкенд — Supabase free tier (Auth + Postgres/RLS + private Storage); хостинг фронта — GitHub Pages. Технические детали — в [ARCHITECTURE.md](ARCHITECTURE.md).

Оценки задач: **S** — до часа, **M** — 1–3 часа, **L** — 3–6 часов (условно, разработка агентная).

## Спринт 1 — «Вход в дом»: каркас, аутентификация, одобрение

**Цель спринта:** развёрнутое на GitHub Pages приложение, в котором можно зарегистрироваться, подтвердить email, дождаться одобрения админа и попасть внутрь. Безопасность работает на уровне БД, а не только UI.

**Результат для пользователя:** «Я захожу на сайт с телефона, регистрируюсь, админ одобряет меня — и я внутри».

### Финальный e2e-чек-лист (прод)

1. Регистрация с телефона нового email → письмо → подтверждение → pending.
2. Одобрение с десктопа из /admin → телефон после обновления внутри.
3. С телефона: добавить персону, подвинуть, связать, загрузить 2 фото, поставить аватар.
4. С десктопа: всё видно; отредактировать биографию; привязать аккаунт к персоне.
5. Выйти → анонимно и pending-аккаунтом убедиться: данные и фото недоступны (в т.ч. прямыми запросами).

### Definition of Done проекта

- Все 4 спринтовых DoD закрыты; e2e-чек-лист пройден на проде.
- В репозитории нет секретов (только публичные URL/anon key в variables).
- README позволяет пересоздать всё с нуля без участия автора.

# Вторая волна (Спринты 5–9) — по итогам аудита

После завершения Спринтов 1–4 проведён полный аудит: живые проверки безопасности против прода и код-ревью всей кодовой базы. Найдены 3 критичные дыры в правах (RLS) и ряд гонок/оптимизаций. ## Спринт 5 — «Замок»: безопасность и код-фиксы

**Результат для пользователя:** «Никто из родни не сможет незаметно присвоить чужую карточку или подменить фото, а мои перемещения кружочков не откатываются».

### Пользовательские истории и критерии

| ID | История | Критерии приёмки |
|---|---|---|
| S5.1 | Как участник, я не могу привязать карточку к чужому аккаунту или подделать автора | INSERT persons с чужим `user_id`/произвольным `created_by` от не-админа отклоняется БД (e2e-тест) |
| S5.2 | Замечания к карточкам защищены от подмены | UPDATE edit_requests не-владельцем меняет только `status`; переписать `message`/`author_id`/`person_id` нельзя |

| S5.5 | Не застреваю на вечном спиннере | Если профиль не пришёл сразу после входа — ретрай/фолбэк, не бесконечная загрузка |

## Спринт 6 — «В кармане»: мобильный UX и PWA

**Цель:** сделать сервис по-настоящему удобным с телефона и устанавливаемым как приложение.

**Результат для пользователя:** «Открываю с телефона, тапаю по кружочку — меню со всеми действиями; добавил приложение на главный экран, оно открывается как родное».

### Пользовательские истории и критерии

| ID | История | Критерии приёмки |
|---|---|---|
| S6.1 | На телефоне тап по кружочку открывает контекстное меню, а не сайдбар | На тач-устройстве одиночный тап → меню с пунктами (первый — «Открыть карточку»); на десктопе поведение прежнее (клик=сайдбар, правый клик=меню) |
| S6.2 | Все действия доступны из меню | Меню: открыть карточку, добавить родственника ▸, связать, редактировать, удалить (админ) |
| S6.3 | Удобно добавлять и ориентироваться на телефоне | FAB-кнопка «+» внизу справа; нижняя мини-панель (экспорт, поиск); кнопки/мишени ≥ 40px |
| S6.4 | Сайдбар удобно закрывать | На мобильном крупная шапка с «← Назад» вместо мелкого × |
| S6.5 | Приложение ставится на главный экран | PWA: manifest, иконка 🌳, standalone-запуск без адресной строки, офлайн-оболочка |

### Задачи

| # | Задача | Оценка |
|---|---|---|
| 6.1 | Детект тач (`matchMedia('(hover: none)')`); на мобильном onNodeClick → контекстное меню с пунктом «Открыть карточку» | M |
| 6.2 | FAB «+» + нижняя мобильная панель действий | M |
| 6.3 | Крупные тач-мишени: меню, крестики связей, кнопки миниатюр; шапка сайдбара «← Назад» | M |
| 6.4 | PWA: vite-plugin-pwa, manifest, иконки, service worker (кэш оболочки) | M |

### DoD спринта
- Проверено на реальном телефоне на проде: тап→меню, FAB работает, установка на главный экран, автономный запуск.
- Десктопное поведение не изменилось.

### Риски
- Service worker может кэшировать старую версию — настроить auto-update и проверить обновление после деплоя.

## Спринт 7 — «Раскладка»: улучшения доски

**Цель:** большие деревья становятся читаемыми и удобными в навигации; каждый видит себя в центре.

**Результат для пользователя:** «Нажал „Разложить как дерево“ — всё встало по поколениям; нашёл прабабушку через поиск; при входе доска сразу на мне».

### Пользовательские истории и критерии

| ID | История | Критерии приёмки |
|---|---|---|
| S7.1 | Я центрируюсь на себе при открытии | Доска при загрузке центрируется на карточке, привязанной к моему аккаунту; если не привязан — общий fitView |
| S7.2 | Я раскладываю дерево одной кнопкой | «Разложить как дерево» — поколения по уровням, супруги рядом, минимум пересечений; после можно двигать вручную, позиции сохраняются |
| S7.3 | Я быстро нахожу человека | Поиск по имени → список → выбор центрирует и подсвечивает ноду |
| S7.4 | Я сохраняю древо картинкой | «Скачать картинкой» → PNG всей доски |
| S7.5 | Я вижу правки родни сразу | Realtime: добавление/перемещение/связь другим участником появляется без F5 |

### Задачи

| # | Задача | Оценка |
|---|---|---|
| 7.1 | Центрирование на своей карточке после первого loadAll (setCenter по user_id) | S |
| 7.2 | Авторасстановка: dagre/elkjs layout по parent-рёбрам, супруги рядом, кнопка + сохранение позиций | L |
| 7.3 | Поиск по имени с центрированием (реюз PersonPickerDialog + setCenter) | M |
| 7.4 | Экспорт в PNG (html-to-image, динамический импорт) | M |
| 7.5 | Realtime-подписка postgres_changes → applyRemoteChange в boardStore; включить Realtime на таблицах | L |
| 7.6 | Оптимизация: kinshipAll/граф без зависимости от pos_*, ленивый лайтбокс/компрессор, React.lazy(BoardPage) | M |

### DoD спринта
- Дерево из 3+ поколений раскладывается кнопкой без пересечений; поиск центрирует; PNG скачивается; правки со второго устройства видны без F5; при входе — своя карточка в центре.

### Риски
- dagre-раскладка семейных графов с супругами нетривиальна — при плохом результате оставить как «черновую», ручная доводка сохраняется.

## Спринт 9 — «Обмен»: GEDCOM и уведомления

**Цель:** интеграция со стандартными генеалогическими сервисами и оповещение админа.

**Результат:** «Выгрузил древо в GEDCOM для MyHeritage; на почту приходит письмо о новой заявке».

### Пользовательские истории и критерии

| ID | История | Критерии приёмки |
|---|---|---|
| S9.1 | Я обмениваюсь древом через GEDCOM | Экспорт persons+relationships в .ged (5.5.1); импорт .ged с маппингом в БД |
| S9.2 | Админ узнаёт о событиях по почте | Письмо при новой заявке на регистрацию и новом замечании (Resend + Edge Function) |

### Задачи

| # | Задача | Оценка |
|---|---|---|
| 9.1 | GEDCOM экспорт (.ged из графа) | M |
| 9.2 | GEDCOM импорт (парсер + вставка с проверкой дублей) | L |
| 9.3 | Edge Function + Resend: уведомления админу (нужен API-ключ Resend от пользователя) | M |

### DoD спринта
- Экспортированный .ged открывается в стороннем сервисе; импорт восстанавливает дерево; письма доходят.

### Риски
- Встроенный SMTP Supabase ограничен — Resend решает; нужен бесплатный аккаунт Resend и ключ.
